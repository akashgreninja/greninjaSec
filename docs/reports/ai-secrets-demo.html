  • aws iam delete-access-key --access-key-id <leaked-key-id> --user-name <IAM-username>
  • aws iam create-access-key --user-name <IAM-username>
✋ Manual Steps
  1. Remove the exposed AWS Access Key ID from all source code, files, and version history (e.g., using git filter-branch or BFG repo-cleaner).
  2. Identify the IAM user associated with the exposed key using the AWS Management Console or CLI.
  3. Delete or deactivate the exposed key immediately in the AWS IAM console or with the AWS CLI.
  4. Rotate to a new access key if required and update all applications/services to use the new key through secure mechanisms (such as environment variables or AWS Secrets Manager).
  5. Invalidate all sessions and monitor for suspicious activity.
  6. Remediate any revealed access secrets as well if any secret key pairs were in the codebase.
🔄 Alternative Solution

If removing the file is not feasible, restrict its permissions and move credentials to a secure vault (e.g., AWS Secrets Manager or HashiCorp Vault), refactoring applications to read from these sources instead of static files.

🧪 Testing Steps
  1. Verify that the exposed key is fully deleted in the AWS IAM console.
  2. Attempt to use the old key and ensure it fails with an 'Access Key Inactive' or 'Access Denied' error.
  3. Check that the new credentials (if rotated) are securely stored and not present in any committed file.
  4. Search repository history for any additional exposures (`git grep AKIA` and `git log -p`).
  5. Review CloudTrail logs for unexpected activity using the compromised key.
